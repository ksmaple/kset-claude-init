# 代码风格规范

## 通用原则
- 保持代码简洁、可读
- 严格遵循项目已有风格和约定
- 不为了风格一致性而大规模重构无关代码
- 拒绝过早抽象：三个相似代码块优于一个假设性的通用工具函数

## 命名规范
- 变量/函数: 有意义的命名，避免单字母（循环变量除外）
- 常量: 全大写蛇形命名 `MAX_COUNT`
- 类/组件: PascalCase
- 文件/目录: kebab-case（除非项目已有其他约定）

## 代码格式
- 使用项目配置的格式化工具（Prettier、rustfmt、black 等）
- 提交前确保代码通过 lint 检查

## 注释规范
- 优先编写自解释代码
- 仅在逻辑复杂、边界情况、非直观实现处添加注释
- 不添加解释"做了什么"的注释，只添加解释"为什么"的注释

## 修改纪律
- 一次改动只解决一个问题
- 修复 bug 时不顺便重构无关代码
- 新增功能时不改变现有未涉及模块的格式
